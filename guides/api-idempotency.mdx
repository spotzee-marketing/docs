Network failures, timeouts, and retries are unavoidable. Spotzee supports idempotency keys so a retried `POST` or `PATCH` returns the original response instead of running the operation twice.

## How it works

<Steps>
  <Step title="Generate a key">
    Generate a unique key per logical request ‚Äî typically a UUID, a hash, or your own primary key.
  </Step>
  <Step title="Send it on the first attempt">
    Add an `Idempotency-Key` header to your `POST` or `PATCH`.
  </Step>
  <Step title="Retry with the same key and body">
    If the request times out or fails partially, retry with the exact same key and exact same body.
  </Step>
  <Step title="Receive the cached response">
    Spotzee replays the original response ‚Äî same status, same body, same `request_id`.
  </Step>
</Steps>

```http
PATCH /users HTTP/1.1
Host: apix.spotzee.com
Authorization: Bearer sk_‚Ä¶
Idempotency-Key: 1f6f4f7c-7a44-4cf6-9b3e-3b8a3a3b3a3a
Content-Type: application/json

[
  { "external_id": "user-9876", "email": "ada@example.com" }
]
```

## Scope and TTL

- **Scope** ‚Äî keys are scoped per API key, so you can reuse the same key string across keys without collision.
- **Per-surface scope** ‚Äî the Main API and Extended API maintain independent caches. The same `Idempotency-Key` sent to both surfaces is treated as two distinct requests; pick a fresh key per surface if you want isolation, or reuse it deliberately knowing each surface will replay independently.
- **TTL** ‚Äî cached responses live for 24 hours. After that, a request with the same key runs as a brand-new operation.
- **Eligible methods** ‚Äî `POST` and `PATCH`. `GET`, `HEAD`, and `DELETE` are idempotent by definition and ignore the header.

## What Spotzee does on replay

| Replay scenario | Outcome |
|---|---|
| Same key, same body | Original response is returned. The `Idempotent-Replayed: true` response header confirms it's a replay. |
| Same key, different body | `409 idempotency_key_mismatch` ‚Äî keys are tied to the exact request body. Use a fresh key for distinct requests. |
| Same key, in-flight (Main API) | `409 idempotency_in_progress` with `Retry-After`. The original request is still running; retry shortly. |
| Same key, in-flight (Extended API) | The duplicate runs and is cached on completion ‚Äî there is no in-flight 409 on this surface. Successive replays after the first response lands return the cached copy. |

## Errors

```json
{
  "status": "error",
  "code": "idempotency_key_mismatch",
  "title": "Idempotency key mismatch",
  "message": "The Idempotency-Key was reused with a different request body. Use a fresh key for distinct requests.",
  "type": "https://docs.spotzee.com/main-api/errors#idempotency_key_mismatch",
  "request_id": "req_01HXY‚Ä¶"
}
```

| HTTP | `code` | When |
|---|---|---|
| `400` | `idempotency_key_invalid` | Empty or longer than 255 characters |
| `409` | `idempotency_key_mismatch` | Same key reused with a different body |
| `409` | `idempotency_in_progress` | Original request still running ‚Äî retry after `Retry-After`. Main API only. |

## Generating keys

Any opaque string up to 255 characters works. Pick a generator that suits your platform:

<CodeGroup>
```javascript JavaScript
crypto.randomUUID()
```

```python Python
import uuid
str(uuid.uuid4())
```

```go Go
import "github.com/google/uuid"
uuid.NewString()
```
</CodeGroup>

## When not to reuse a key

Reuse a key for the same logical operation ‚Äî for example, retrying the same "create user" request after a timeout. Mint a fresh key when:

- The user submits the form a second time after deciding to change a field.
- You're processing a queue and each message is a distinct operation.
- You're running a backfill ‚Äî each row is its own logical request.

## Caching rules

- `2xx` responses are cached.
- `4xx` responses are cached (your retry would have failed the same way).
- `5xx` and `429` responses are not cached so retries can succeed when the platform recovers.
